Getting Started with the Workspace ONE UEM API

Workspace ONE UEM provides a collection of RESTful APIs that allow external programs to use the core product functionality by integrating the APIs with existing IT infrastructures and third-party applications. Leveraging the simplified REST style of software architecture, Workspace ONE UEM REST APIs currently support a multitude of functionalities, including organization group, console administration, mobile application, mobile device, email, enrollment user, profile, smart group, and user group management.

Leveraging REST-based APIs provide several benefits to enterprises, including eliminated cost and time spent developing applications in-house. Workspace ONE UEM REST APIs are fully able and ready to integrate with enterprise servers, programs, and processes. Additionally, Workspace ONE UEM REST APIs are more efficient, can run smoothly and can be easily branded with enterprises. These APIs are intended for application developers, and this guide provides an understanding of design and architecture of Workspace ONE UEM's API library to facilitate custom development and integration with Workspace ONE UEM.

To begin leveraging the APIs, REST API access must be enabled in the Workspace ONE UEM. For help setting up the console, please see our Initial Setup page. Client applications can then request a service document located at the API application entry point, e.g. https://{host}/api/system/info. Dereferencing this URL will yield a root service document describing which resources are available to the caller.

Version Remotion

Workspace ONE UEM has modified the look of the API URLs by removing the version specification from them. For example: Assign Smart Group to a Public Application

Old API URI:

https://host/api/v1/mam/apps/public/{applicationid}/addsmartgroup/{smartgroupid}

New API URI:

https://host/api/mam/apps/public/{applicationid}/smartgroup/{smartgroupid}

Different API versions can be fetched by adding the version details in the Accept header. A single API URL serves the purpose of retrieving multiple versions based on the version details in the Accept header. Below is an example of the Accept header to fetch version 2 of an API.

Accept=application/json;version=2
OR
Accept=application/xml;version=2

Workspace ONE UEM will also return the API version used as well as a list of all the supported versions of the API in the response headers. The 'x-api-version' header specifies which version of the API was used, and the 'api-supported-version' header lists the supported versions of the API. For example, if version 1 of an API was requested, but the API supports both version 1 and 2, the response headers will be:

x-api-version: v1
api-supported-version: 1,2

Important Note: If no version is sent in the Accept header, version 1 of the API will be returned by default. If a version higher than the maximum supported version is requested, the latest version of the API will be used.

API Rate Limits

Workspace ONE UEM limits the number of API requests that can be made during certain time periods. Rate limiting is done per Organization Group (OG) based on the API key used in the request. There are two different rate limits:

Server Throttling - The limit imposed on an OG for a 1 minute interval.
Daily Quota - The limit imposed on an OG for a 24-hour interval.

Each additional API key created at an OG will aggregate to the total Server Throttling and Daily Quota set for that OG.

The response headers from an API call contain information about the rate limits set on the APIs. Below are examples of the headers and an explanation of what they represent.

x-ratelimit-limit: 50000 - The total number of API calls that can be made in a 24 hour period
x-ratelimit-remaining: 48505 - The remaining number of API calls that can be made in the 24 hour period
x-ratelimit-reset: 1503683551 - The exact time when the rate limit will reset. The value is in epoch time format.

Important API Considerations

Authentication

Workspace ONE UEM supports multiple types of authentication for APIs. The authentication credentials are sent in the "Authorization" HTTP request header.

  • Basic Authentication - Base64-encoded username and password.
  • Certificate Authentication - Signature of either the request message or URL, signed with an admin certificate generated in the Workspace ONE UEM.
  • OAuth Authentication - Bearer token aquired from an identity provider, such as VMware Identity Manager.

Workspace ONE UEM APIs also require an API key to be sent for additional authorization. This key is sent in the "aw-tenant-code" request header. The key can be found on the REST API settings page of the Workspace ONE UEM.

aw-tenant-code: (The key value is available in the system settings when REST API access is enabled.)

Endpoint URIs

Client applications MUST NOT make any assumptions about resource URI layouts, structure or hierarchy as these are controlled by the server and can change over time.

SSL only

Workspace ONE UEM requires all requests to be made over SSL.

Device Identifier Formats

Workspace ONE UEM supports five types of device identifiers:

  • MAC address
  • Serial Number
  • UDID
  • Device Id
  • EAS Id

Media Types

Request/response bodies and resource representations are encoded as either XML or JSON. Each type of resource has its own media type, or MIME type, which matches the pattern application/http://www.air-watch.com/{resourcetype}+{format} where {resourcetype} represents a unique representation of a resource (e.g. a device) and {format} is either xml or json (text/xml or application/json).

Important Note: The API must provide a representation of all resource available as either XML or JSON. The API must allow client applications to use application/xml or application/json as a synonym for the default representation at a given resource URL.

Http Status Codes

Workspace ONE UEM API clients MUST be ready to accept HTTP status codes as defined in the following list:

1xx:Informational - Communicates transfer protocol-level information

100 "Continue"
This indicates that the client can continue with its request.
101 "Switching Protocols"
This indicates that the protocol version or protocol is being changed.

2xx:Success - Indicates that the client request was accepted successfully

200 "OK"
This indicates nonspecific success. In most cases, 200 is the code the client hopes to see. It indicates that the Workspace ONE UEM API successfully carried out the requested action, and that no other status code in the 2xx series is appropriate. Unlike the 204 status code, a 200 response should include a response body. 200 ("OK") is not used to communicate errors in the response body.
201 "Created"
This indicates successful resource creation. The Workspace ONE UEM API responds with the 201 status code whenever a collection creates a new resource.
202 "Accepted"
This indicates successful start of an asynchronous action. A 202 response indicates that a request will be handled asynchronously. This response status code indicates that the request appears valid, but it may still have problems after being processed. A 202 response is typically used for actions that take a long while to complete. Controller resources may send 202 responses, but other resource types should not.
203 "Non Authoritative Information"
This indicates that the returned meta information is from a cached copy instead of the origin server and therefore may be incorrect.
204 "No Content"
Used to indicate that the response body is intentionally empty. The 204 status code is usually sent in response to a PUT, POST, or DELETE request when the Workspace ONE UEM API declines to send back any status message or representation in the response message body. The API may also send 204 in conjunction with a GET request to indicate that the requested resource exists, but has no state representation to include in the body.
205 "Reset Content"
This indicates that the client should reset (not reload) the current resource.
206 "Partial Content"
This indicates that the response is a partial response as requested by a GET request that includes a byte range.

3xx:Redirection - Indicates that the client must take additional action to complete its request

300 "Multiple Choices"
This indicates that the requested information has multiple representations. The default action is to treat this status as a redirect and follow the contents of the Location header associated with this response.
300 "Ambiguous"
This indicates that the requested information has multiple representations. The default action is to treat this status as a redirect and follow the contents of the Location header associated with this response.
301 "Moved Permanently"
This indicates that a resource has been relocated. The 301 status code indicates that the Workspace ONE UEM API's resource model has been significantly redesigned and a new permanent URI has been assigned to the requested resource. The Workspace ONE UEM API should specify the new URI in the response's Location header.
301 "Moved"
This indicates that the requested information has been moved to the URI specified in the Location header. The default action when this status is received is to follow the Location header associated with the response. When the original request method was POST,the redirected request will use the GET method.
302 "Found"
This indicates that the requested information is located at the URI specified in the Location header. The default action when this status is received is to follow the Location header associated with the response. When the original request method was POST, the redirected request will use the GET method.
302 "Redirect"
This indicates that the requested information is located at the URI specified in the Location header. The default action when this status is received is to follow the Location header associated with the response. When the original request method was POST, the redirected request will use the GET method.
303 "See Other"
This is used to refer the client to a different URI. A 303 response indicates that a controller resource has finished its work, but instead of returning a potentially unwanted response body, it returns the URI of a response resource. This can be the URI of a temporary status message or to a more permanent existing resource. Generally speaking, the 303 status code allows the Workspace ONE UEM API to send a reference to a resource without forcing the client to download its state. Instead, a GET request may be sent to the value of the Location header.
303 "Redirect Method"
This automatically redirects the client to the URI specified in the Location header as the result of a POST. The request to the resource specified by the Location header will be made with a GET.
304 "Not Modified"
This is used to preserve bandwidth. This status code is similar to 204 ("No Content") in that the response body must be empty. The key distinction is that 204 is used when there is nothing to send, whereas 304 is used when there is state information associated with a resource but the client already has the most recent version of the representation. This status code is used in conjunction with conditional HTTP requests, e.g. If-Modified header.
305 "Use Proxy"
This indicates that the request should use the proxy server at the URI specified in the Location header.
306 "Unused"
This is a proposed extension to the HTTP/1.1 specification that is not fully specified.
307 "Temporary Redirect"
This indicates that the request must be resubmited to another URI. HTTP/1.1 introduced the 307 status code to reiterate the originally intended semantics of the 302 ("Found") status code. A 307 response indicates that the Workspace ONE UEM API will not process the request. Instead, the request should be resubmitted to the URI specified by the response's Location header. The Workspace ONE UEM API may use this status code to assign a temporary URI to a requested resource. For example, a 307 response can be used to shift a request over to another host.
307 "Redirect Keep Verb"
This indicates that the request information is located at the URI specified in the Location header. The default action when this status is received is to follow the Location header associated with the response. When the original request method was POST, the redirected request will also use the POST method.

4xx:Client Error - Indicates that there was a problem on the client side

NOTE: For errors in the 4xx category, the response body may contain a document describing the error (unless the request method was "HEAD").
400 "Bad Request"
This indicates nonspecific failure. 400 is the generic client-side error status, used when no other 4xx error code is appropriate.
401: "Unauthorized"
This is used when there is a problem with the client's credentials. A 401 error response indicates that the client tried to operate on a protected resource without providing proper authorization. It may have provided the wrong credentials or none at all.
402: "Payment Required"
PaymentRequired is reserved for future use.
403: "Forbidden"
This is used to forbid access regardless of authorization state. A 403 error response indicates that the request is formed correctly, but the Workspace ONE UEM API refuses to honor it. A 403 response is not a case of insufficient client credentials (which would be 401 ("Unauthorized")). The Workspace ONE UEM API uses 403 to enforce application-level permissions. e.g. a client may be authorized to interact with some, but not all, Workspace ONE UEM API resources. If the client attempts a resource interaction outside of its permitted scope, a 403 will be returned.
404 "Not Found"
This is used when the requested URI cannot be mapped to a resource. The 404 error response indicates that the Workspace ONE UEM API can't map the requested URI to a resource.
405 "Method Not Allowed"
This is used when the HTTP method is not supported. A 405 error response indicates that a request used an HTTP method that the resource does not allow. For instance, a read-only resource might support only GET and HEAD, while a controller resource might allow GET and POST, but not PUT or DELETE. A 405 response must include the Allow header, which lists supported HTTP methods.
406 "Not Acceptable"
This is used when the requested media type cannot be served. A 406 error response indicates that the API is not able to generate any of the preferred media types as indicated by the Accept request header. e.g. a request for data formatted as txt/csv will receive a 406 response if the API is only willing to format the data as application/json.
407 "Proxy Authentication Required"
This indicates that the requested proxy requires authentication. The Proxy-authenticate header contains the details of how to perform the authentication.
408 "Request Timeout"
This indicates that the client did not send a request within the time the server was expecting the request.
409 "Conflict"
This is used to indicate a violation of resource state. A 409 error response indicates that the client tried to put Workspace ONE UEM API resources into an impossible or inconsistent state. e.g. The API may return this status code when a client tries to delete a non-empty store resource.
410 "Gone"
This indicates that the requested resource is no longer available.
411 "Length Required"
This indicates that the required Content-length header is missing.
412 "Precondition Failed"
This is used to support conditional operations. Clients may specify one or more preconditions in its request headers, effectively telling the API to process the request only if certain conditions were met. A 412 response indicates that those conditions were not met.
413 "Request Entity Too Large"
This indicates that the request is too large for the server to process.
414 "Request Uri Too Long"
This indicates that the URI is too long.
415 "Unsupported Media Type"
This is used when a request's media type payload cannot be processed. A 415 error response indicates that the API is unable to process the supplied media type, as indicated by the Content-Type request header. e.g. a client request including data formatted as txt/html will receive a 415 response if the API is only willing to process data formatted as application/json.
416 "Requested Range Not Satisfiable"
This indicates that the range of data requested from the resource cannot be returned, either because the beginning of the range is before the beginning of the resource, or the end of the range is after the end of the resource.
417 "Expectation Failed"
This indicates that an expectation given in an Expect header could not be met by the server.
426 "Upgrade Required"
This indicates that the client should switch to a different protocol such as TLS/1.0.

5xx:Server Error - Indicates that there was a problem on the server side

500 "Internal Server Error"
This indicates an error with the API. A 500 error response is the generic API error response that is returned whenever it encounters an exception.
501 "Not Implemented"
This indicates the service does not currently support the requested functionality.
502 "Bad Gateway"
This indicates that an intermediate proxy server received a bad response from another proxy or the origin server.
503 "Service Unavailable"
This indicates the server is unable to handle the request due to temporary overload, or to maintenance.
504 "Gateway Timeout"
This indicates that an intermediate proxy server timed out while waiting for a response from another proxy or the origin server.
505 "Http Version Not Supported"
This indicates that the requested HTTP version is not supported by the server.
